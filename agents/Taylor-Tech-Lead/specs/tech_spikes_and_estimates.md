요약

이 문서는 Product spec(초기_product_spec.md)의 제약(중앙값 150ms @ 10k RPS, 인프라 예산 제한, 기존 인증 사용)을 바탕으로 수행할 기술 스파이크(technical spike) 목록, 각 스파이크의 목적·가정·수행방법·수행시간 추정(예상 인력 포함), 성공/종료 기준(acceptance criteria)을 정리합니다.

스파이크 목록 (우선순위: P1 → P4)

A. API 디자인·성능 검증 스파이크 (P1)
- 목적: 현재 제안된 API 설계가 10k RPS 조건에서 median latency 150ms 목표를 만족하는지 검증. 병목(네트워크, JSON 직렬화, DB 쿼리)을 식별.
- 가정: 기존 인증 사용(오프로드 가능), 서비스는 FastAPI 기준으로 성능 테스트.
- 산출물: 부하 테스트 시나리오(k6/locust 스크립트), 측정 리포트(latency p50/p95/p99, error rate), 최적화 권고안
- 작업 항목:
  1) 테스트 엔드포인트 선정(핵심 API 5개) 및 데이터 샘플 준비
  2) 부하 테스트 스크립트 작성(k6/locust)
  3) 기본 배포(스테이징) + 리소스 설정
  4) 점증적 부하(1k → 10k RPS) 테스트 및 데이터 수집
  5) 병목 식별 및 1차 최적화(쿼리 튜닝, 캐시 레벨 추가)
- 필요 인원/기간: 백엔드 엔지니어 1명(풀타임) + 인프라 엔지니어 0.5명, 5 업무일
- Exit Criteria: 10k RPS에서 median latency ≤ 150ms 또는 성능 장애 원인과 개선 작업 목록(우선순위 포함)

B. DB 성능·스키마 검증 스파이크 (P1)
- 목적: 예상 쿼리 패턴에 대해 Postgres가 견딜 수 있는지, 인덱스·파티셔닝·레플리카 설계 필요성을 판정
- 산출물: 쿼리 프로파일, 인덱스 제안서, 파티셔닝/레플리카 아키텍처, 비용/운영 영향
- 작업 항목:
  1) 핵심 쿼리 목록 수집(읽기/쓰기 비율 포함)
  2) TPC-like 데이터셋(또는 샘플 실데이터 마스킹) 준비
  3) 벤치마크(psql, pgbench, custom scripts)
  4) 인덱스·파티션·복제 실험 및 결과 분석
- 필요 인원/기간: DB/백엔드 엔지니어 1명, 4 업무일
- Exit Criteria: 95% 요청 패턴에서 p50/p95 목표 만족 가능한 DB 구성 제안서

C. Autoscaling·비용 모델링 스파이크 (P1)
- 목적: 예산 제약 내에서 안정적으로 10k RPS를 처리할 수 있도록 오토스케일 정책(임계치, min/max 인스턴스)을 검증하고 비용 모델을 제시
- 산출물: 권장 오토스케일 정책, 비용 시나리오(베이스/피크/버스트), SLO 대비 비용 그래프
- 작업 항목:
  1) 인스턴스 성능 프로파일(한 인스턴스가 안정적으로 처리 가능한 RPS 추정)
  2) 스케일 정책 설계(CPU, latency, request-per-second-per-instance, custom metric)
  3) 시뮬레이션/실측으로 비용 산정
- 필요 인원/기간: 인프라 엔지니어 1명 + 백엔드 0.5명, 3 업무일
- Exit Criteria: 명확한 min/max, scale up/down 조건, 월간 예산 시나리오(낮/평균/최대)

D. 재사용 가능한 백엔드 컴포넌트 PoC (P2)
- 목적: 공통으로 사용 가능한 미들웨어·컴포넌트(레이트 리미터, 캐시, 토큰 버킷, 큐)를 검증하고, 라이브러리화·배포 방식 제안
- 산출물: PoC 코드(간단한 모듈), 통합 가이드, 예상 적용 비용/운영 부담
- 작업 항목:
  1) Redis 기반 캐시·rate-limiter PoC
  2) DB connection pool 설정 검증
  3) 메시지 큐(RQ/Redis, RabbitMQ) 간단 연동
- 필요 인원/기간: 백엔드 1명, 4 업무일
- Exit Criteria: PoC로 핵심 기능 동작 확인 및 적용 가이드

리스크·의존성
- 기존 인증 시스템 의존: 인증 호출 지연이 전체 latency에 영향 줄 수 있음 → 스파이크에서 auth latency 포함 테스트 필요
- 인프라 예산 미정/타이트: max 인스턴스 제한 시 성능 목표 불가할 수 있음 → 비용 시나리오가 반드시 필요

우선 권장 실행 순서
1) A (API 부하) + C (오토스케일 모델링)를 병행 (데이터 수집 상호 보완)
2) B (DB) — A 결과에 따라 우선순위 조정
3) D (재사용 컴포넌트) — 초기 PoC는 A/B 수행 중 병렬 진행 가능
